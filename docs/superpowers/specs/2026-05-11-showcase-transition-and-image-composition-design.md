# Showcase 过渡与图像构图编辑设计

## 概述

当前项目已经具备可工作的 showcase 页面、editor 页面、双语 editor UI，以及基础的滚动激活区块机制。但在实际体验中仍存在几个明显问题：

- 区块切换时背景变化过于生硬
- 滚动到两个区块的边界时，背景有来回闪回现象
- `hero` 区块的视觉仍偏向“框内图片”，海报感不足
- 其他带单图展示的区块缺少足够直观的视觉编辑能力
- 现有标签（`tags`）虽然会渲染，但 editor 中没有对应的编辑入口

本次设计的目标，是在不引入重型动画库的前提下，对 showcase 的切换机制、hero 呈现方式、单图布局控制能力以及 tag 编辑体验进行一轮成体系的升级。

## 目标

- 让区块过渡保留氛围感，但明显减少闪回与抖动
- 让 `hero` 区块改为真正的背景图主视觉，而不是依赖卡片化图片框
- 让非 gallery 的单图区块支持更强的视觉编排能力
- 让 editor 提供直观的图像控制方式，包括滑条与右侧预览中的直接拖动
- 让 editor 补齐 tags 的新增、删除与排序能力
- 保持 `#/` 展示页为只读展示，避免将编辑交互泄漏到公开展示页

## 非目标

- 不为 gallery 多图区块增加每张图独立裁切/拖动编辑
- 不引入重型动画运行时或第三方拖拽大框架
- 不把公开展示页改成可交互编辑器
- 不在本轮中重做整个数据模型或引入 CMS 式结构
- 不改变 editor 中双语 locale 的整体方向

## 问题定义

### 1. 区块切换抖动

当前 active section 的判定依赖 `IntersectionObserver` 回调内，对当前 entries 做 `intersectionRatio` 排序并直接选取最高值。这个策略在正常滚动时足够简单，但在两个区块边界附近容易出现：

- 上一个区块和下一个区块交替成为“最高可见比率”
- active id 短时间反复切换
- 背景样式、遮罩和内容态来回闪动

### 2. 背景过渡生硬

当前背景色/渐变由 `ShowcaseCanvas` 根节点直接根据 active section 切换，虽然有基础 `transition`，但背景图层与内容图层没有真正分离，导致视觉上更像“整个页面样式瞬切”，而不是“海报氛围逐步切换”。

### 3. hero 呈现不足够海报化

当前 `hero` 与其他单图区块共享主要的图片呈现路径，图像更多作为正文结构里的一个可见元素，而不是整屏主视觉背景。结果是：

- hero 的空间占用不够大
- 文案与主视觉之间缺少层级压迫感
- 整体更像一个漂亮的内容块，而不是海报首屏

### 4. 单图区块缺乏可控构图

当前单图区块图片基本是静态嵌入，缺少：

- 画框大小控制
- 图片缩放控制
- 在画框内的直接取景/定位控制

因此用户很难把图片真正调整到理想构图。

### 5. tags 没有编辑入口

`SectionConfig.tags` 已存在于数据模型中，也会被 renderer 渲染，但 editor 里没有对应的 tag 编辑控件。这使得该字段虽然存在，但实际不可维护。

## 设计方向

### 视觉方向

已确认的视觉取向如下：

- 过渡以“保留氛围感”为优先，而不是极度保守地只求稳定
- active section 判定要偏稳一些，但不是迟钝到拖慢切换反馈
- 背景切换更适合做柔和混合，而不是简单瞬切
- `hero` 风格偏干净：以暗色渐变遮罩为主，不走明显大光斑路线
- 文字可读性优先，通过遮罩保障，而不是再套重卡片结构

### 交互方向

- editor 左侧继续提供数值/滑条型控制
- editor 右侧真实预览中允许直接拖动单图区块图片位置
- 公开展示页 `#/` 只读展示最终结果，不提供拖动交互
- tags 的编辑方式采用更贴近真实 tag editor 的交互，而不是逗号分隔纯文本框

## 方案比较

### 方案 A：仅调 active 判定阈值

只通过修改 `IntersectionObserver` 的 threshold、root、排序条件、切换阈值来减少闪回。

**优点：**
- 改动最小
- 可以较快改善闪回

**缺点：**
- 不能解决 hero 不够海报化的问题
- 不能解决单图区块缺乏构图控制的问题
- 只修现象，不改善整体视觉体系

### 方案 B：背景层与内容层分离 + hero 背景化 + 单图区块可控构图（推荐）

将过渡问题、hero 主视觉问题、单图区块图片编辑问题合并设计：

- 单独抽象背景视觉层
- 稳定 active section 判定
- `hero` 背景化
- 其他单图区块支持画框、缩放、拖动取景
- tags 在 editor 中可维护

**优点：**
- 能一次解决当前最主要的体验痛点
- 设计上更统一
- editor 的可编辑能力明显提升

**缺点：**
- 涉及数据模型、renderer、editor、CSS、交互测试多个层面
- 比单点修阈值复杂

### 方案 C：直接引入动画/拖拽库

通过第三方动画系统与拖拽系统强行覆盖当前逻辑。

**优点：**
- 某些局部效果可能更快做出来

**缺点：**
- 引入额外复杂度
- 与当前项目“轻量、静态托管友好”的目标不一致
- 不必要的依赖成本较高

### 推荐方案

采用 **方案 B**。

理由：它既能解决当前实际体验问题，又能把图片编辑能力补到一个更符合“可复用展示模板”的程度，同时仍然保持在纯前端、轻量可控的架构里。

## 数据模型扩展

建议在 `SectionConfig` 上新增一组服务于单图呈现与 hero 背景化的字段：

```ts
imageDisplayMode: "inline" | "background"
imageFrameWidth: number
imageFrameHeight: number
imageScale: number
imageOffsetX: number
imageOffsetY: number
```

### 字段含义

- `imageDisplayMode`
  - `background`：图片作为区块背景主视觉
  - `inline`：图片仍作为区块内部的独立可见区域

- `imageFrameWidth`
  - 单图区块图片容器宽度控制值
  - 建议归一化为百分比或基于固定范围的数值

- `imageFrameHeight`
  - 单图区块图片容器高度控制值

- `imageScale`
  - 图片在容器中的缩放比例

- `imageOffsetX`
  - 图片在容器中的水平偏移

- `imageOffsetY`
  - 图片在容器中的垂直偏移

### 适用范围

- `hero`：强制使用 `background`
- `profile` / `custom` / 其他未来单图区块：通常使用 `inline`
- `gallery`：本轮不接入这套单图控制模型

### 兼容性与默认值

新字段需要在以下位置补默认值与 runtime 归一化：

- 默认配置创建函数
- `createSection()`
- `normalizeSiteConfig()` / `normalizeSection()`
- 导入/导出与本地持久化读取流程

这样可以保证：
- 旧存储数据仍可安全回退
- 新字段即使缺失，也不会破坏布局

## 渲染设计

### 1. 背景层与内容层分离

`ShowcaseCanvas` 不应只依赖根节点背景值变化来完成区块切换。

建议将区块背景视觉拆为更明确的层次：

- 基础背景渐变层
- hero 背景图层（当 active section 为背景模式时生效）
- 暗色渐变遮罩层
- 可选的轻微装饰层（但第一版不强调）

这样在 active section 切换时，可以让：

- 背景图或背景渐变独立平滑过渡
- 内容层继续用现有的 active/idle class 过渡
- 避免所有视觉元素绑死在一次 class 切换上

### 2. active section 判定稳定化

当前的“取最大 intersectionRatio”策略需要升级为更稳的切换模型。

建议原则：

- 不因为两个区块在边界附近轻微交错就立刻切换
- 更倾向在新区块明确进入主视区后再切 active
- 当当前 active 仍有足够可见度时，尽量维持当前 active

实现层面不要求复杂状态机，但应具备“偏稳”的滞后切换逻辑。

### 3. hero 背景化

`hero` 区块中的 `image` 应优先作为背景主视觉使用：

- 图片铺在 hero 背景层中
- 文案压在其上
- 通过暗色渐变遮罩保障文字可读性
- hero 不再依赖一个独立的“图片框”作为主视觉中心

### 4. 其他单图区块的 inline 呈现

对于 `profile`、`custom` 等单图区块：

- 保持图片区块存在
- 但其宽高比例、内部图片缩放与偏移都可配置
- 通过 CSS + transform 实现取景与裁切

## Editor 交互设计

### 1. hero 遮罩控制

在 editor 的 Selected section 面板中，当 section 为 `hero` 时，应出现直观的遮罩控制：

- 遮罩强度 slider（基于 `overlayOpacity`）

必要时后续可以扩展 hero 背景缩放，但第一版优先保证：

- hero 背景化
- 遮罩强度可调

### 2. 单图区块图片控制

对于非 gallery 的单图区块，在 editor 中增加：

- 画框宽度 slider
- 画框高度 slider
- 图片缩放 slider

这些控件负责调整：

- `imageFrameWidth`
- `imageFrameHeight`
- `imageScale`

### 3. 右侧预览直接拖动定位

在 `#/editor` 右侧嵌入预览中，当当前选中的区块属于可拖动单图区块时：

- 用户可以在图片可视区域中拖动图片
- 拖动结果实时写回 `imageOffsetX` / `imageOffsetY`
- 拖动只在 editor 预览中启用
- 公开展示页 `#/` 只读取最终结果，不可交互拖动

这是本轮最重要的“直观编辑”能力。

### 4. tags 编辑器

`tags` 的编辑方式采用可维护的 tag editor，而不是逗号分隔字符串。

第一版建议支持：

- 新增 tag
- 删除 tag
- 拖动排序

不需要一开始就做复杂的自动补全或 tag 建议系统。

目标是让 `tags: string[]` 在 UI 层真正可编辑、可排序、可维护。

## 样式与体验

### 1. hero 文案可读性

hero 文案不应直接裸压在图片上，而应借助：

- 暗色半透明渐变遮罩
- 合理的文字区域布局
- 必要的阴影/对比度增强

重点是“干净但可读”，而不是强烈光斑氛围。

### 2. 单图区块图框手感

画框大小变化时，应避免布局完全塌陷或跳变得太难看。建议：

- 限定最小/最大宽高范围
- 让图框调整更偏向“构图微调”而不是完全自由布局破坏

### 3. 编辑状态可感知

在 editor 里，支持拖动定位的图片区块最好在 hover / active / dragging 时有轻微状态反馈，例如：

- 边框提示
- cursor 变化
- 拖动时的轻微高亮

但第一版不需要过度复杂化。

## 错误处理与边界

- 没有图片的区块不显示图片构图控件
- gallery 区块不显示单图拖动/缩放控件
- 旧配置导入时缺少新字段，自动回退到默认值
- 若用户把图片缩放或偏移调到极端值，仍应保证渲染安全，不出现 NaN 或完全崩坏
- 拖动交互只在 editor 嵌入预览启用，公开展示页不附带编辑手柄或可拖状态

## 测试策略

建议至少覆盖：

1. 新字段默认值与 runtime normalization
2. hero 背景化渲染逻辑
3. 单图区块画框尺寸渲染逻辑
4. 单图区块拖动定位更新逻辑
5. tags 新增 / 删除 / 排序的 editor 交互
6. active section 稳定化后，不再在边界处频繁来回切换（至少做局部逻辑级测试）

对于真正的“滚动体验”与“拖动手感”，仍需浏览器手测确认。

## 验收标准

当以下条件全部满足时，认为本轮改造完成：

- hero 区块图像以背景主视觉方式呈现
- 区块边界滚动时，背景不再明显来回闪回
- 区块切换整体更柔和，不再显得过于生硬
- 非 gallery 单图区块可通过 editor 调整画框宽高与图片缩放
- 在 editor 右侧预览中，可直接拖动单图区块图片位置
- 拖动结果刷新后仍能保留
- tags 可新增、删除、拖动排序
- `#/` 展示页只读展示最终结果，不包含拖动编辑交互
