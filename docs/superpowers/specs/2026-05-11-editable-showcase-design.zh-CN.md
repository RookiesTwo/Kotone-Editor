# 可编辑展示页设计

## 概述

构建一个基于 GitHub Pages 部署的纯前端项目，技术栈使用 Vite 和 React。项目包含一个高视觉冲击的滚动驱动展示页，以及一个配套编辑页，使用户无需修改代码即可重新配置几乎全部内容。

最终产物应同时满足以下两种定位：

- 一个适合 AI 项目实践提交的高完成度示例站点
- 一个可复用的、面向未来角色主题落地页的可编辑框架

## 目标

- 创建一个视觉表现强、内容极简的落地页，节奏和氛围参考 GTA 6 官网
- 保持内容模型通用化，使同一个项目可以复用于其他角色或主题
- 通过独立路由支持展示与编辑两种工作流
- 将编辑结果存储到 `localStorage`
- 支持 JSON 导入与导出，便于备份、迁移和提交演示
- 允许用户在编辑页中新增、删除、隐藏、排序并编辑每个区块
- 提供一个高质量的默认示例主题，同时保留通用框架结构
- 以纯静态前端方式平滑部署到 GitHub Pages

## 非目标

- 不做后端、数据库、登录认证或 CMS
- 不做拖拽式低代码搭建器
- 不使用重型动画引擎，也不做复杂 3D 特效
- 不要求用户为日常内容修改而直接编辑源码

## 技术栈

- Vite
- React
- TypeScript
- React Router，用于 `/` 和 `/editor`
- 使用原生 CSS、CSS 变量和过渡效果处理主题与动效

本项目优先使用 TypeScript，因为它更适合这种“配置驱动”的应用结构，能够为展示页、编辑页、存储层以及导入导出流程提供清晰的数据契约。

同时，技术选型只是指导原则，不应凌驾于实现质量之上。比起机械坚持某项技术，更重要的是项目是否清晰、稳定、易于完成和易于交付。如果某些局部实现采用更轻量的方式能明显提升可读性或可靠性，这种取舍是允许的，只要整体架构仍然保持一致和干净。

## 产品结构

应用包含两个主要路由：

- `/`：公开展示页
- `/editor`：带实时预览的配置编辑页

展示页完全由一个统一配置对象 `siteConfig` 驱动。

编辑页负责读取和更新同一个配置对象。所有持久化、导入导出、预览和渲染行为都应围绕这套共享数据模型展开。

## 体验概览

### 展示页

展示页应更像一个滚动驱动的视觉落地页，而不是普通的长文章页面。

核心特征：

- 强主视觉图像
- 极少文案
- 超大字号排版
- 深色基底配合高饱和强调色
- 大留白与海报式构图
- 随滚动触发的分屏视觉过渡

页面由多个可配置的全屏或接近全屏的区块组成。当激活区块切换时，页面应平滑过渡背景、文本强调、叠层以及前景图像等视觉元素。

### 编辑页

编辑页应优先强调可用性，而不是装饰性。

布局：

- 左侧为控制面板
- 右侧为实时预览

预览区应尽量贴近真实展示效果，让每次编辑都能立即体现意义。

## 数据模型

使用一个根对象：

```ts
type SiteConfig = {
  global: GlobalConfig
  sections: SectionConfig[]
}
```

TypeScript 主要用于定义应用内部的数据模型，但对于导入的 JSON 和本地持久化恢复的数据，仍然需要运行时校验。

### 全局配置

`global` 应覆盖站点级设置，例如：

- 站点标题
- 站点副标题
- 主题强调色
- 背景基础色
- 字体风格预设
- 外部链接
- 可选的导航或页脚显示控制
- 动效强度或过渡预设

### 分屏配置

`sections` 中的每一项代表一个屏幕或一个主要视觉区块。

每个区块应支持：

- 稳定的 `id`
- `type`，例如 `hero`、`profile`、`gallery`、`quote`、`custom`
- `visible`
- `title`
- `description`
- `tags`
- `image`
- 可选的第二张图片或装饰素材
- 背景色或渐变
- 遮罩强度
- 对齐方式预设
- 布局密度或高度预设
- 可选按钮文案和链接
- 区块级过渡样式元数据

编辑器必须允许用户：

- 新增区块
- 删除区块
- 调整区块顺序
- 切换显示/隐藏
- 修改区块类型
- 编辑所有支持的字段

## 区块类型

框架应默认支持以下区块类型：

- `hero`：主开场屏，带主视觉图片和超大标题
- `profile`：简短的角色或主题介绍区块
- `gallery`：包含一张或多张插画的视觉区块
- `quote`：极简语句或高亮文案区块
- `custom`：面向后续复用的灵活图文区块

系统应设计成后续可以新增区块类型，而不需要重写整体结构。

## 资源来源

图片字段应同时支持：

- 远程图片 URL
- 适用于 GitHub Pages 部署的本地静态资源路径

如果图片缺失或无效，界面应平滑降级，可以显示占位效果，或隐藏失效视觉元素。

## 滚动与过渡行为

展示页应以“当前激活区块”作为主要状态驱动。

推荐交互模型：

- 在滚动过程中检测当前激活区块
- 通过 React 状态和 CSS 类切换激活态与非激活态
- 使用 CSS 过渡处理透明度、位移、颜色和遮罩变化

目标效果是受控的“翻页感”：

- 背景会随区块切换而变化
- 图片通过轻微位移或缩放平滑进出
- 标题和副标题通过分层动效进入
- 遮罩和装饰形状改变透明度或位置

边界约束：

- 不做重型 3D 场景
- 除非确有必要，不引入大型第三方动画依赖
- 优先保证静态托管稳定性和可接受的移动端性能

## 编辑流程

编辑页应提供四组主要控制：

### 全局设置

- 标题
- 副标题
- 主题配色
- 字体预设
- 外部链接
- 共享布局元素的可选显示开关

### 区块管理

- 新建区块
- 删除区块
- 调整区块顺序
- 显示或隐藏区块
- 修改区块类型

### 当前区块设置

- 文本内容
- 标签
- 按钮与链接
- 图片
- 背景与遮罩设置
- 对齐与密度预设
- 过渡相关设置

### 数据操作

- 保存到 `localStorage`
- 重置为内置示例配置
- 导出 JSON
- 导入 JSON

编辑器可以自动保存，也可以提供手动保存并给出明确反馈。无论采用哪种方式，刷新后都应恢复最近一次有效保存的数据。

## 持久化与导入导出

持久化策略：

- 首次加载时使用内置示例配置
- 如果 `localStorage` 中存在已保存数据，则优先使用它
- 允许用户重置回内置示例配置

导入导出行为：

- 将完整 `siteConfig` 导出为 JSON
- 导入时必须先验证数据结构，再决定是否替换当前配置
- 对无效 JSON 或无效配置结构应给出明确失败提示，并保持当前数据不变

## 默认示例内容

框架应内置：

- 一套可复用的通用内容结构
- 一个高质量示例主题，用于展示预期视觉标准

这个示例应足够完整，能够直接用于报告或演示，而无需额外准备。

## 错误处理

应优先采用防御性回退，而不是脆弱的硬失败。

预期规则：

- 缺少保存数据时回退到默认配置
- 导入的 JSON 格式错误时不覆盖当前配置
- 空文本字段不应破坏布局
- 空链接字段应隐藏对应按钮
- 缺失图片时应显示安全的回退效果
- 未知区块类型应安全跳过，或显示基础占位内容

## 响应式行为

项目必须同时支持桌面端和移动端。

桌面端优先级：

- 更具电影感的视觉构图
- 更强的区块切换过渡
- 更宽松的大间距布局

移动端优先级：

- 保留相同的主题气质和层级关系
- 在必要时降低动效强度
- 避免文字溢出和布局堆叠错乱
- 即使空间较紧，也要保证编辑页可用

## 部署

项目目标部署平台为 GitHub Pages。

部署要求：

- 仅输出静态构建产物
- base path 配置必须兼容 GitHub Pages
- 本地资源路径在部署后仍可正常访问
- 路由设计需要考虑静态托管下 `/` 与 `/editor` 的访问方式

## 验证要求

在认为实现完成之前，至少验证以下内容：

- 本地开发服务器可正常运行
- 生产构建成功
- 展示页路由渲染正常
- 编辑页路由渲染正常
- `localStorage` 保存和刷新恢复行为正常
- JSON 导出功能正常
- 有效 JSON 可以正确导入
- 无效 JSON 导入时能安全处理
- 区块新增、删除、排序和显示控制正常
- 桌面端和移动端布局均可用
- 滚动驱动过渡不会造成明显布局损坏或严重卡顿

## 推荐架构

代码组织应围绕几个职责清晰的模块展开：

- 应用壳层与路由
- 共享配置 schema 与默认示例数据
- 存储与导入导出工具
- 展示页渲染器与分屏组件
- 编辑器界面与配置表单控件
- 主题与过渡样式

核心架构原则是：展示页必须由配置驱动渲染，而不是写死页面内容。

## 选择该方案的原因

这个设计刻意避开两个极端：

- 不是一个写死内容、一次性使用的静态页面
- 也不是一个会吞噬项目范围的过度复杂可视化编辑器

最终选用的是一个平衡型可编辑框架：

- 足够精致，适合展示
- 足够通用，便于复用
- 足够简单，能够作为纯前端项目完成
- 足够完整，便于在 AI Agent 开发实践与 GitHub Pages 部署报告中清晰说明
